Overview

The DEVSEOPS Enterprise Branching Model defines a structured, traceable Git workflow designed for teams using JIRA for issue tracking and a continuous delivery pipeline. It enforces clear separation between development, release, and emergency hotfix lifecycles.

BranchSourceTarget (MR)PurposeLifecycle
master —— Production-ready source of truth Permanent
feature/* mastermaster Feature development per JIRA ticket Short-lived · Delete after merge
release tag master— Immutable release snapshot Permanent tag on master
hotfix/* release tagmaster Emergency production fix Short-lived · Delete after merge
hotfix release tag hotfix/*— Versioned hotfix release snapshot Permanent tag on hotfix

Step-by-Step Workflow
1

Create Feature Branch from Master

All development starts by branching off master. Branch must be named using the JIRA ticket number: feature/PROJ-1234

2

Raise MR: feature/* → master

Once development is complete and tested locally, raise a Merge Request from feature/* into master. Peer review required before merge.

3

Create Release Tag from Master

After MR is merged, create an immutable annotated tag on master: v1.0.0.0. This marks the production release candidate.

4

Create Hotfix Branch from Release Tag

For production bugs, branch off the specific release tag. Name it hotfix/v1.0.0.1 and bump the 4th version digit (BUILD).

5

Create Release Tag from Hotfix

Once the hotfix is validated, tag the hotfix branch: v1.0.0.1. This becomes the new production release that gets deployed.

6

Merge Hotfix MR back to Master

Raise MR from hotfix/* into master to ensure the fix is in the main codebase. Prevents future regression.

⚠️
Critical: Hotfix branches must always be merged back to master (Step 6). Skipping this step causes master to diverge from production and leads to regressions in future releases.

Naming Conventions
TypePatternExampleRule
feature feature/<JIRA-TICKET> feature/PROJ-1234 JIRA ticket number is mandatory. Use uppercase project key.
hotfix hotfix/v<X.X.X.X> hotfix/v1.2.0.1 Version must match the release tag being patched, with last digit incremented.
release tag v<MAJOR.MINOR.PATCH.BUILD> v1.2.0.0 4-part versioning. Always use annotated tags (not lightweight).
hotfix tag v<X.X.X.BUILD+1> v1.2.0.1 Increment the BUILD digit for each hotfix on the same MINOR version.

Git Commands — Branch Operations
🌿 Feature Branch Workflow
  bash — Feature Branch Flow
# STEP 1: Ensure master is up-to-date
git checkout master
git pull origin master

# STEP 2: Create feature branch from master
git checkout -b feature/PROJ-1234
# modern alternative:
git switch -c feature/PROJ-1234

# STEP 3: Develop, commit your changes
git add .
git commit -m "feat(PROJ-1234): add user authentication module"

# STEP 4: Push feature branch to remote
git push origin feature/PROJ-1234

# STEP 5: Sync feature branch with latest master (before raising MR)
git fetch origin
git rebase origin/master

# STEP 6: After MR approval and merge — clean up branch
git branch -d feature/PROJ-1234
git push origin --delete feature/PROJ-1234
🔍 Branch Listing & Inspection
  bash — Branch Inspection
# List all local branches
git branch

# List all remote branches
git branch -r

# List all local + remote branches with last commit info
git branch -avv

# Show only feature branches
git branch --list "feature/*"

# Show only hotfix branches
git branch --list "hotfix/*"

# Branches already merged into master (safe to delete)
git branch --merged master

# Branches NOT yet merged into master
git branch --no-merged master

# Show remote tracking details
git remote show origin

Git Commands — Tag Operations
📌 Creating & Pushing Release Tags
  bash — Release Tagging
# TAG FROM MASTER (normal release) ─────────────────────────────────

# Ensure you're on master and up-to-date
git checkout master
git pull origin master

# Create ANNOTATED tag (recommended — includes tagger, date, message)
git tag -a v1.0.0.0 -m "Release v1.0.0.0 — Initial production release"

# Push specific tag to remote
git push origin v1.0.0.0

# Push ALL local tags at once
git push origin --tags

# Tag a specific commit by SHA
git tag -a v1.0.0.0 <commit-sha> -m "Release v1.0.0.0"

# TAG FROM HOTFIX BRANCH ────────────────────────────────────────────
git checkout hotfix/v1.0.0.1
git tag -a v1.0.0.1 -m "Hotfix v1.0.0.1 — Fix critical null pointer [PROJ-5678]"
git push origin v1.0.0.1
🔍 Tag Listing & Inspection
  bash — Tag Inspection
# List all tags
git tag

# List tags matching a pattern
git tag -l "v1.*"

# Show tag details — annotated message + commit info
git show v1.0.0.0

# Checkout source at a specific tag (detached HEAD)
git checkout v1.0.0.0

# Create a hotfix branch FROM a specific release tag
git checkout -b hotfix/v1.0.0.1 v1.0.0.0

# Delete a local tag (use with caution)
git tag -d v1.0.0.0

# Delete a remote tag (use with extreme caution — coordinate with team)
git push origin --delete v1.0.0.0

Git Commands — Log & History
  bash — Log & History Reference
# BASIC LOGS ──────────────────────────────────────────────────────────

# One-line visual graph — best for branch overview
git log --oneline --graph --all --decorate

# Full commit detail with author, date, subject
git log --format="%H %an %ae %ad %s" --date=short

# FILTERED LOGS ───────────────────────────────────────────────────────

# Commits on a specific branch
git log feature/PROJ-1234 --oneline

# Commits between two release tags (useful for changelog generation)
git log v1.0.0.0..v1.0.0.1 --oneline

# Commits by a specific author
git log --author="Jane Doe" --oneline

# Commits touching a specific file (with full diff)
git log --follow -p src/auth/login.py

# Find commits referencing a JIRA ticket
git log --grep="PROJ-1234" --oneline

# Commits since a date
git log --since="2026-01-01" --oneline

# BRANCH DIFF TRACKING ────────────────────────────────────────────────

# What's in feature/* but not yet in master
git log master..feature/PROJ-1234 --oneline

# What's in hotfix/* but not yet in master
git log master..hotfix/v1.0.0.1 --oneline

# Full pretty graph (save as git alias)
git log --graph --pretty=format:'%Cred%h%Creset -%C(yellow)%d%Creset %s %Cgreen(%cr) %C(bold blue)<%an>%Creset' --abbrev-commit --all

# RECOMMENDED GIT ALIASES (add to ~/.gitconfig)
# [alias]
#   lg  = log --oneline --graph --all --decorate
#   lga = log --graph --pretty=format:'%Cred%h%Creset -%C(yellow)%d%Creset %s %Cgreen(%cr)' --abbrev-commit --all

Git Commands — Full Hotfix Workflow
🚨
Emergency Use Only. Hotfix branches are reserved for critical production defects that cannot wait for the next normal release cycle. Coordinate with the Release Manager before creating a hotfix branch.
  bash — Complete Hotfix Flow (Steps 4 through 6)
# STEP 4: Create hotfix branch from release tag ─────────────────────
git fetch origin --tags
git checkout -b hotfix/v1.0.0.1 v1.0.0.0
# Branch is now at the exact state of release v1.0.0.0

# Bump version in your code (version.txt / pom.xml / package.json)
# Change: 1.0.0.0  ->  1.0.0.1
git add version.txt
git commit -m "chore: bump version to v1.0.0.1"

# Apply the fix
git add .
git commit -m "fix(PROJ-5678): resolve critical NPE in auth service"

# Push hotfix branch to remote
git push origin hotfix/v1.0.0.1

# STEP 5: Tag the hotfix branch ─────────────────────────────────────
git tag -a v1.0.0.1 -m "Hotfix v1.0.0.1 — Fix critical NPE in auth [PROJ-5678]"
git push origin v1.0.0.1

# STEP 6: MR hotfix/* -> master (raise via GitLab/GitHub UI)
# After MR approval and merge, verify on master:
git checkout master
git pull origin master
git log --oneline -10

# Clean up hotfix branch
git branch -d hotfix/v1.0.0.1
git push origin --delete hotfix/v1.0.0.1

# Hotfix complete. v1.0.0.1 is in production. master is fully synced.

Branch Protection Rules & Governance
  • ✓
    master is a protected branch — direct pushes are not allowed. All changes must come via an approved MR.
  • ✓
    All MRs to master require at least 1 peer reviewer approval before merge.
  • ✓
    CI/CD pipeline must pass all checks (build, test, security scan) before MR can be merged.
  • ✓
    Feature branches must be deleted after merge to keep the repository clean.
  • ✓
    Branch name must follow feature/JIRA-XXXX or hotfix/vX.X.X.X convention — enforced via push rules.
  • ✓
    Release tags are immutable and permanent — never delete or force-move a release tag.
  • ✓
    Hotfix branches must always be merged back to master (Step 6) to prevent regression in future releases.
  • ✓
    Version bump commit must be the first commit on a hotfix branch, before applying the fix.
  • ✓
    All commit messages must reference the JIRA ticket: fix(PROJ-5678): ... or feat(PROJ-1234): ...
✅
DSO Compliance: Teams following this model achieve full traceability from JIRA ticket → commit → branch → release tag → production deployment, satisfying enterprise audit and change-management requirements.
