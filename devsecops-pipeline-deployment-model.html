1. Overview

This document defines the enterprise-grade DevSecOps Pipeline Deployment Model governing how code progresses from developer workstations through all environments to production. The model enforces security scanning, automated quality gates, and controlled release management aligned with the organisation's change-management and compliance requirements.

ℹ️

Scope

Applies to all application repositories hosted in the enterprise Git platform. Teams must not bypass pipeline stages without an approved Change Request (CR).

3. Branch Strategy

The pipeline follows a trunk-based GitFlow variant with four branch categories, each with distinct deployment targets, quality gates, and approval requirements.

3.1  feature/* — Feature Development Branch

💡

Purpose

Short-lived branches created from master for individual features, stories, or tasks. Naming convention: feature/<JIRA-ID>-short-description (e.g. feature/PROJ-421-payment-gateway).

AttributeDetail
Sourcemaster
Deploy TargetDEV  (Optional — developer's discretion)
TriggerManual pipeline trigger or push event (configurable per team)
Gates RequiredUnit tests · SAST scan · Code style lint
LifecycleDeleted after merged Pull Request is approved into master
Merge StrategySquash commit via PR with minimum 1 peer review
  • Create Branch

    Branch off master: git checkout -b feature/PROJ-421-payment-gateway

  • Develop & commit

    Push commits; each push triggers CI (compile, unit test, SAST). Optional DEV deploy available.

  • Open Pull Request → master

    Required: passing CI, SonarQube quality gate, ≥1 approver. Branch is then merged and deleted.

3.2  master — Integration Branch

✅

Purpose

The primary integration branch. All feature and hotfix work is merged here. Every commit to master automatically triggers the full QA pipeline (QAF → QAP → QAR) before a release tag is created.

StageEnvironmentGate TypePass Criteria
CI Build DEV Automated Compile success · unit test pass · Docker image build
Security Scan Pipeline Automated SAST (SonarQube) · SCA (OWASP Dependency Check) · Secret scan
QA Functional (QAF) QA Automated All integration/E2E test suites green, <1% failure rate
QA Performance (QAP) QA Automated p95 response ≤ SLA threshold · no memory leak detected (JMeter/k6)
QA Regression (QAR) QA Automated Full regression suite 100% pass · zero P1/P2 defects
Tag Creation — Manual (Release Manager) CAB approval · release notes attached

3.3  tag — Release Tag (Deployment Trigger)

🏷

Purpose

An immutable, signed Git tag created from master after all QA gates pass. The tag is the sole trigger for deployments to UAT, UAT-DR, PROD, and PROD-DR. No direct deployments to these environments are permitted without a valid tag.

AttributeDetail
Source Branchmaster only
Naming (Normal)1.0.0   Semantic Versioning — MAJOR.MINOR.PATCH
Naming (Hotfix)1.0.0-1   Base version + sequential hotfix counter
SigningGPG-signed tags required (git tag -s)
Deployment OrderUAT → UAT-DR ↔ PROD → PROD-DR
Approval GateCAB (Change Advisory Board) sign-off required before PROD deploy
RollbackRe-deploy previous tag; no code changes required

Deployment sequence on tag push:

  • UAT Deploy

    Deploy tagged artefact to UAT. Run automated smoke tests and UAT sign-off by business stakeholders.

  • UAT-DR Replication

    Sync deployment to UAT Disaster Recovery site. Validate DR database replication and health checks.

  • PROD Deploy

    After CAB approval, deploy to Production using blue/green or rolling strategy. Monitor alerts for 30 min.

  • PROD-DR Replication

    Sync deployment to Production DR site. Full failover test executed quarterly.

3.4  hotfix/* — Emergency Fix Branch

🚨

Purpose

Emergency branches created from master (at the current PROD tag) when a P1 defect is detected in production. Hotfix branches must validate lower environments before creating a hotfix release tag. After deployment, the branch is merged back into master.

AttributeDetail
Source Branchmaster (at the commit of the failing PROD tag)
Naming Conventionhotfix/<INCIDENT-ID>-short-description e.g. hotfix/INC-8842-payment-timeout
Deploy TargetsDEV → QA (QAF + QAR)
QA ScopeFunctional (QAF) + Regression (QAR) — Performance (QAP) skipped for speed
Tag Version1.0.0-1   (suffix increments for each hotfix: 1.0.0-1, 1.0.0-2, …)
Post-Deploy MergeHotfix branch merged back into master; next normal release resets to 1.1.0
SLA TargetP1 hotfix from branch creation to PROD deploy ≤ 4 hours

4. Versioning Model

Version TypeExample TagSource BranchWhen Used
Major Release 2.0.0 master Breaking API changes or major architectural changes
Minor Release 1.1.0 master New features, backwards-compatible
Patch Release 1.0.1 master Bug fixes, no new features
Hotfix Release 1.0.0-1 hotfix/* Emergency P1/P2 production fix; suffix counter increments per hotfix
Subsequent Hotfix 1.0.0-2 hotfix/* Second hotfix applied against the same base version before next minor release
📌

Versioning Rule

The hotfix suffix (-N) resets to -1 whenever the MAJOR.MINOR.PATCH component increments. After merging all hotfixes into master, the next tag from master must bump at least the PATCH segment (e.g. 1.0.0 → 1.0.1).

6. QA Gate Definitions

🧪 QAF — Functional Testing

Validates application behaviour against functional requirements. Includes integration tests, API contract tests, and end-to-end UI flows.

Tools: Selenium · Playwright · Postman / Newman · Cucumber

⚡ QAP — Performance Testing

Validates non-functional requirements: throughput, response times, concurrency, and resource utilisation under simulated production load.

Tools: Apache JMeter · k6 · Gatling · Grafana

🔄 QAR — Regression Testing

Full regression suite executed to ensure new changes have not introduced regressions into existing functionality. Runs the complete test library covering all application modules.

Tools: Test management platform · automation suite · defect tracking (Jira)

7. Deployment Flows

▶ Normal Release Flow (master → tag → UAT → PROD) 
# 1. Merge feature PRs into master
git checkout master
git merge --no-ff feature/PROJ-421

# 2. Push triggers CI + QA pipeline automatically
#    QAF → QAP → QAR must all pass

# 3. Release Manager creates GPG-signed tag
git tag -s 1.0.0 -m "Release 1.0.0 – Q2 Feature Bundle"
git push origin 1.0.0

# 4. Tag push auto-triggers pipeline:
#    Deploy UAT → UAT-DR → [CAB APPROVAL] → PROD → PROD-DR
▶ Hotfix Release Flow (hotfix/* → validate → tag → PROD) 
# 1. P1 incident raised; hotfix branch from master
git checkout master        # at commit matching failing PROD tag
git checkout -b hotfix/INC-8842-payment-timeout

# 2. Apply fix, push → auto-triggers DEV deploy + QAF + QAR
git commit -m "fix: increase payment gateway timeout to 30s"
git push origin hotfix/INC-8842-payment-timeout

# 3. All lower-env gates pass; Release Manager creates hotfix tag
git tag -s 1.0.0-1 -m "Hotfix 1.0.0-1 – Payment timeout fix INC-8842"
git push origin 1.0.0-1

# 4. Tag push auto-triggers: UAT → UAT-DR → PROD → PROD-DR

# 5. Merge hotfix back into master (no squash — preserve history)
git checkout master
git merge --no-ff hotfix/INC-8842-payment-timeout
git push origin master

# 6. Delete hotfix branch
git push origin --delete hotfix/INC-8842-payment-timeout
