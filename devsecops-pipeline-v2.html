1. Overview

This document defines the end-to-end DevSecOps CI/CD Pipeline covering all branch types, pipeline stages, registry promotion, environment deployments, and security controls. The pipeline applies to both containerised applications (deployed to OpenShift) and non-containerised applications (deployed to datacentres via Ansible).

ℹ️

Scope

All application repositories on the enterprise Git platform. No direct deployments to any environment are permitted without a passing pipeline. No direct pushes to master — all changes must go through a Merge Request (MR).

4. Registry Promotion Strategy

Two distinct registries govern the image lifecycle. Images are never rebuilt for higher environments — the same image is promoted from the Snapshot Registry to the Release Registry. This guarantees that what is tested in QA is exactly what is deployed to PROD.

5. Branch Policies

5.1  feature/* — Feature Development Branch

💡

Purpose

Short-lived branches for individual features or tasks. Created from master. Example: feature/PROJ-421-payment-gateway.

AttributeDetail
Deploy TargetDEV — OpenShift   (on successful pipeline)
Pipeline StagesBuild → Test → Security (SAST, SCA, SonarQube, SBOM) → Publish → Build Image → Aqua Scan → Push to Artifactory → Deploy OpenShift DEV
Registry PushImage pushed to Artifactory Registry (not the snapshot registry)

MR ApprovalMinimum 1 peer reviewer; committer cannot be the approver (Segregation of Duty)
Merge StrategySquash commit; branch deleted after merge

5.2  master — Integration Branch

✅

Purpose

The main integration branch. Receives all feature merges. Every MR merge triggers the full CI pipeline automatically, followed by QA deployment. A separate release pipeline creates the release tag once QA is certified.

  • MR merge auto-triggers CI Pipeline

    Build · Test · Security (SAST, SCA, SonarQube, SBOM) · Publish · Build Image · Aqua Scan

  • Push image to Snapshot Registry

    Image is tagged with the Git commit SHA (e.g., app:abc1234f) and pushed to the Snapshot Registry. This guarantees traceability back to the exact commit.

  • Deploy to OpenShift QA

    The snapshot image is deployed to the QA environment. QA team runs full functional, performance, and regression suites.

  • QA Sign-off

    QA team certifies the build. Once certified, the Release Manager executes the Release Pipeline.

  • Release Pipeline: Create Tag + Promote Image

    The release pipeline creates a GPG-signed Git tag (e.g., 1.0.0), promotes the snapshot image to the Release Registry with the version tag, and automatically raises an MR to bump the version number in the codebase (e.g., update pom.xml or package.json).

  • Release Tag triggers deployment pipeline

    The tag push auto-triggers: Deploy UAT → UAT-DR Sync → CAB Approval Gate → Deploy PROD → PROD-DR Sync.

5.3  tag — Release Tag

🏷

Purpose

Immutable, GPG-signed tags are the sole deployment trigger for UAT and PROD. No environment above QA can be deployed to without a valid release tag.

5.4  hotfix/* — Emergency Fix Branch

🚨

Purpose

Emergency branches created from the failing release tag (not from master HEAD). Used when a P1/P2 defect is found in production. The fix is validated in lower environments before a hotfix release tag is created.

6. Security Controls (DevSecOps)

ControlToolStagePolicy on Failure
Static Application Security Testing (SAST) Checkmarx / SonarQube SAST CI Pipeline — Security stage Block pipeline on Critical or High findings
Software Composition Analysis (SCA) OWASP Dependency-Check / Snyk CI Pipeline — Security stage Block on CVSS ≥ 7.0 with no available fix
Code Quality Gate SonarQube CI Pipeline — Security stage Block pipeline and MR merge on quality gate failure
Software Bill of Materials (SBOM) Syft / CycloneDX CI Pipeline — Security stage Generate and attach to artefact; alert on policy violation
Secret Scanning GitLeaks / TruffleHog Pre-commit hook + CI Security stage Block commit and pipeline; alert Security team
Container Image Scanning (Aqua Scan) Aqua Security After Build Image stage Block image push on Critical CVE in OS packages or app dependencies
Infrastructure as Code Scan Checkov / tfsec CI Pipeline — Security stage Block on policy violations (e.g., privileged containers, exposed secrets)
Image Signing Cosign / Notary Push to Release Registry Unsigned images rejected; deployment blocked
CAB Approval Gate ITSM / ServiceNow CR Pre-PROD deploy Pipeline halted until approved Change Request is in place
⚠️

Aqua Scan — Container-Specific Control

Aqua Security scans every container image after it is built, checking for OS-level CVEs, misconfigured packages, compliance policy violations, and runtime risk. Any image with a Critical CVE is blocked from being pushed to any registry until the vulnerability is remediated or an approved exception is granted.

7. Segregation of Duty & Access Control

RuleEnforcementRationale
Committer cannot approve their own MR Git platform branch protection: self-approval blocked; minimum 1 independent approver required Prevents untested or malicious code reaching master without peer review
No direct push to master Branch protection rule: direct pushes rejected; force-push blocked All changes must pass CI pipeline and peer review before integration
Release tag creation is restricted Only the Release Manager role can create and push tags; enforced by Git permissions Prevents unauthorised or premature releases to production environments
PROD deploy requires CAB approval Pipeline gate blocks PROD deploy until an approved Change Request (CR) exists in ITSM Ensures change management compliance and stakeholder awareness
Release Registry is write-protected Only the Release Pipeline service account can push to the Release Registry Ensures only promoted, QA-approved images reach UAT and PROD

8. Container vs. Non-Container Deployment

The pipeline supports both containerised applications (deployed to OpenShift) and traditional applications (deployed to datacentres via Ansible). The first four stages are identical. The deployment path diverges at the Publish stage.

🔀 Container vs Non-Container Pipeline Paths
ℹ️

Non-Container Process

For non-containerised applications, the pipeline is identical through the Publish stage. After the package is published to Artifactory, deployment uses Ansible playbooks to pull the versioned artefact and deploy to the target datacentre servers. The same branch rules, registry promotion concepts (snapshot → release in Artifactory), and approval gates apply.

📌

Version Bump MR

After every release tag is created, the Release Pipeline automatically raises an MR to update the version in the codebase (e.g., pom.xml, package.json, Chart.yaml) to the next SNAPSHOT or development version. This ensures the master branch always reflects the next planned version.
